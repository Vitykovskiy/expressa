# Slice A Runtime And Validation Path (Issue #6)

This document is the devops handoff for implementation issue `#6`.

## Scope

- versioned runtime manifest for Slice A services (backend + postgres)
- VPS deployment script for Slice A runtime
- CI flow that validates compose, executes a build step, and runs smoke/e2e entrypoints

## Repository Assets

- Workflow: `.github/workflows/slice-a-devops-validation.yml`
- Runtime compose: `infra/staging/docker-compose.slice-a.yml`
- VPS deployment script: `infra/vps/deploy-slice-a-runtime.sh`
- Smoke entrypoint: `infra/validation/run-smoke.sh`
- E2E entrypoint: `infra/validation/run-e2e.sh`
- Smoke runner build file: `infra/validation/Dockerfile.smoke`

## Runtime Variables

- `BACKEND_IMAGE` (required for deploy)
- `POSTGRES_PASSWORD` (required)
- `ADMIN_TELEGRAM_ID` (required)
- `DISABLE_TG_AUTH` (required)
- `POSTGRES_DB` (optional, default `expressa`)
- `POSTGRES_USER` (optional, default `expressa`)
- `BACKEND_PORT` (optional, default `18081`)
- `BACKEND_INTERNAL_PORT` (optional, default `80`)
- `DATABASE_URL` (optional; autogenerated when omitted)

## CI Contract

`slice-a-devops-validation.yml` performs:

1. shell syntax validation for runtime/validation scripts
2. compose rendering validation for Slice A runtime manifest
3. image build step (`infra/validation/Dockerfile.smoke`)
4. local runtime up/down with smoke execution
5. e2e entrypoint invocation (non-strict mode until issue `#7` contributes executable scenario assets)

## Operational Notes

- The validation workflow uses `traefik/whoami` in CI to verify the runtime path mechanics before backend image publication.
- Production and staging deployment should pass an actual backend image reference to `deploy-slice-a-runtime.sh`.
- When issue `#7` delivers executable e2e assets, set `E2E_STRICT=true` in the calling context so missing assets fail the pipeline.
